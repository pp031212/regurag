# 概览型 / 分情形问题评测样本设计

这份文档的目标不是立刻改主链路，而是先补一组能验证“系统是否会把总括型问题拆成不同适用情形来回答”的评测样本。

当前判断：

- 这类能力现在不是主线最高优先级
- 但应该尽快补进评测，避免一直默认“用户都会问得很具体”
- 先用评测确认问题是否真实存在，再决定是否升级 retrieval / answer planning

---

## 一、这组样本要测什么

不是测“能不能召回一条最像的规则”，而是测：

1. 用户问的是总括性问题时，系统能否召回多种适用情形
2. 如果不同情形适用不同条文，系统能否避免只答其中一种
3. 如果知识库只覆盖部分情形，系统能否明确说出边界，而不是补全不存在的规则
4. 最终回答是否按情形组织，而不是把多条规则混成一句模糊结论

---

## 二、建议新增一个独立评测集

建议不要先塞进现有 `real_kb` 主样本集，而是先单独建一份：

- `backend/evals/rag_eval_overview_cases.jsonl`

原因：

- 这组题的失败含义和普通 grounded/no_answer 不完全一样
- 它更偏“多情形覆盖”和“按情形归纳”
- 后续更适合单独统计一个 `overview_coverage_hit`

---

## 三、建议样本类型

### 类型 A：总括性规则题

典型形式：

- “X 一般怎么处理？”
- “X 有哪些情况？”
- “X 怎么办？”

这类题需要系统主动把不同适用情形拉出来。

### 类型 B：同一主题下多条不同适用规则

典型形式：

- “辞退员工必须满足哪些条件？”
- “不发工资怎么办？”

这类题不能只答一个条文，而要至少覆盖主要分支。

### 类型 C：总括问题 + 部分情形缺失

典型形式：

- 问题看起来像要总览
- 但知识库只覆盖其中两种情况

这类题要测系统会不会乱补。

## 五、这组题怎么判定更合理

这组题不建议只看 `answer_hit`，建议多看 3 个层面。

### 1. Retrieval / final context 有没有覆盖多情形

核心问题：

- 检索是不是只抓了一条最像的条文
- final context 里有没有至少 2 到 3 条不同情形规则

### 2. 回答有没有“分情形组织”

核心问题：

- 是否有“试用期 / 正式期 / 特殊情形”这类分点
- 或至少明确说“需要区分不同情形”

### 3. 有没有乱补不存在的规则

这是最重要的反面检查：

- 知识库没写的，不要替它补全
- 如果只覆盖部分情形，要明确说边界

---

## 六、建议先做轻量人工判分，不要急着全规则化

这组 overview case，初期不建议全靠严格关键词规则判死。

更现实的做法：

- 先跑 retrieval / final_context 命中
- 再人工抽查 5 到 10 条回答
- 看问题到底主要出在：
  - 没召回多情形
  - 召回了但没组织好
  - 生成时乱概括

这样才能知道下一步该改哪层。

---

## 七、如果这组题真的掉点，下一步改造顺序

建议不要一上来做重型 answer planner，先按下面顺序迭代：

1. 先补 overview query 识别
2. 再扩 final context，保留更多相关条文
3. 再在生成提示里要求“按情形分点说明”
4. 只有这三步还不够，再考虑单独的 answer planning

也就是说：

> 先做 retrieval / final-context 层的轻升级，再考虑生成层结构化规划。

---

## 八、和你当前项目的关系

你当前项目已经有一些这方面的雏形：

- `is_overview_query(...)`
- `collect_overview_article_parents(...)`
- `collect_related_restriction_article_parents(...)`
- `build_final_context_parents(...)`

这些机制说明系统已经意识到：

> 概览型问题不能只答单条最高分结果。

但它还不是完整通用能力。

所以这组评测样本的价值就在于：

- 先验证当前雏形够不够用
- 再决定要不要把“分情形回答”升级成显式能力

---

## 九、一句话结论

> 先补这组 overview case，再决定要不要做“按情形归纳回答”的能力升级；不要在没有评测证据前，直接把系统做重。
