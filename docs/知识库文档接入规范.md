# 知识库文档接入规范

## 适用范围

本规范用于 `ReguRAG` 的知识库文档上传与预处理环节。目标不是让系统“尽量吃下所有格式”，而是提高入库后的切块质量、检索命中率和引用可追溯性。

一句话原则：

- 系统支持上传，不代表推荐原样入库。
- 越接近结构清晰、语义连续的文本，问答效果越稳定。

## 推荐直接入库的格式

- `.txt`
  - 适合规章条款、说明文本、问答整理稿。
- `.md`
  - 适合已经按标题层级和条款结构整理过的制度文档。
- `.docx`
  - 适合正文、表格、附件说明都较规整的制度材料。
- `.xlsx`
  - 适合字段边界明确、列含义稳定的表格型资料。

推荐前提：

- 标题、条款、编号和段落关系清楚。
- 同一段内容尽量围绕一个主题或规则。
- 文档里没有大量无关页、宣传页、图片海报或扫描页。

## 支持上传但建议先整理的格式

- 文本型 `.pdf`
  - 仅在排版规整、复制文本质量较高时建议直接上传。
- 含少量截图或扫描章的 `.docx`
  - 当前已支持 `DOCX` 内嵌图片 `OCR`，但图片越多，处理稳定性和引用质量越容易下降。
- 单张图片或少量图片
  - 适合临时补充截图类规则，不适合作为大批量制度资料的主入口。

建议先做的整理：

- 去掉封面、目录、宣传页、签章页等无问答价值的页面。
- 把连续截图、长截图拆成更清晰的文本说明。
- 对 OCR 容易误识别的表格、印章、竖排内容先做人肉校对。

## 不建议直接入库的内容

- 扫描件占比高的 `PDF`
- 长截图、海报式图片、流程图拼图
- 页眉页脚干扰严重、分栏复杂、跨页断裂严重的 `PDF`
- 信息密度低、主要靠视觉排版表达关系的材料
- 未清理的附件打包文本，例如：
  - 大量重复通知
  - 无关说明页
  - 多份制度混在同一个文件里但没有明确分隔

这类内容即使能被系统处理，也往往只代表“可以吃进去”，不代表“适合进入知识库主链”。

## 上传前检查清单

上传前，至少过一遍下面这几项：

1. 去掉与问答无关的封面、空白页、宣传页、页脚说明和重复附件。
2. 把复合规则拆开，尽量让一段文字只描述一个主体、条件、流程或处理结果。
3. 补齐主语、适用对象、触发条件、时间边界和例外情形，减少“这个、那种、上述情况”之类指代。
4. 如果原文主要靠表格、截图或扫描图承载信息，优先整理成连续文本再上传。
5. 一份文档尽量只服务一个明确主题；不要把多个业务域的制度长期混放在同一个文件里。

## 预处理建议

### 规章制度类

- 保留条款编号和标题。
- 尽量把“情形 -> 处理方式 -> 例外说明”写在相邻文本里。
- 不要只保留摘要而删掉关键限定词。

### 表格类

- 确保列标题明确，避免合并单元格过多。
- 如果表格表达的是规则，最好补一段自然语言说明。

### OCR 类材料

- 先确认文字方向正常、图片清晰、无大面积遮挡。
- 对容易误识别的人名、数字、日期、分数区间做一次人工复核。

## 当前非目标

`ReguRAG` 当前是文本主导的垂直制度知识问答系统，不以“最大化兼容所有原始格式”为目标。上传链路的重点是：

- 让知识库内容更适合切块和检索
- 让引用能回到可核对的原文
- 让高风险制度问题尽量建立在清晰文本证据上

因此，文档预处理和入库质量控制，优先级高于继续扩更多低价值格式。
