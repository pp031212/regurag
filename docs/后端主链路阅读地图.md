# 后端主链路阅读地图

这份文档不是设计总览，而是给人读代码用的“导航图”。

目标只有两个：

1. 先知道一次问答请求从哪里进、到哪里出
2. 再知道哪些是主链路，哪些是外围治理，哪些是影子链路

---

## 1. 先记住 3 个入口

平时读后端，只先看这 3 个入口：

- `backend/app/services/rag_service.py::RAGService.query()`
- `backend/app/rag/pipeline.py::RAGPipeline.ask()`
- `backend/app/workflows/rag/graph.py::build_shadow_graph()`

可以先这样理解：

- `RAGService.query()`：服务层总入口，负责路由、短路、守卫、持久化
- `RAGPipeline.ask()`：核心问答流水线，负责检索、选上下文、生成答案
- `build_shadow_graph()`：核心流水线的影子图实现，用来做对照，不负责整条服务链

---

## 2. 一次请求的主路径

最常见的正常问答路径如下：

1. 前端请求进入 `RAGService.query()`
2. 做知识库路由、会话解析、轻意图/FAQ/离题短路判断
3. 如果没有被短路，进入 `RAGPipeline.ask()`
4. `ask()` 先准备查询输入，再调用 `retriever.retrieve()`
5. 检索结果进入 MMR、parent 聚合、rerank、final context 构建
6. 调 LLM 生成答案
7. 回到 `RAGService.query()`，补路由 debug、answer guard、no-answer 清理
8. 持久化 user / assistant message 和 `message_context`
9. 返回结果

如果只想抓主线，脑子里记住一句话就够：

`RAGService.query() -> RAGPipeline.ask() -> retriever.retrieve() -> rerank/generate -> persist`

## 3. `RAGService.query()` 到底负责什么

代码位置：

- [rag_service.py](/D:/System/Desktop/regurag/backend/app/services/rag_service.py:684)

它不是“RAG 算法主体”，而是**服务层编排器**。

它主要负责 5 类事情：

### 3.1 路由和请求落点

- 自动知识库路由：`KnowledgeBaseRouter`
- 校验目标知识库是否存在、是否 ready
- 解析/创建会话

### 3.2 不进主链路的短路分支

这些分支命中后，不会进入 `pipeline.ask()`：

- cross-domain guard
- light intent
- FAQ shortcut
- off-topic / meaningless input

所以如果你发现一次请求根本没跑检索，先看这里是不是提前 return 了。

### 3.3 会话历史解析

- `ConversationContextResolver`
- 负责把当前问题和历史消息整理成 `history_rewritten_query`

注意：

- 这一步属于**服务层外围能力**
- 它不在 shadow graph 里

### 3.4 调用核心 pipeline

真正进入核心问答的地方是：

- [rag_service.py](/D:/System/Desktop/regurag/backend/app/services/rag_service.py:867)

这里会把这些信息传给 `pipeline.ask()`：

- query
- top_k_retrieve
- top_k_rerank
- enable_rewrite
- enable_rerank
- answer_style
- `query_prep`
- `history_rewritten_query`

### 3.5 结果后处理和持久化

`pipeline.ask()` 返回之后，这里还会继续做：

- shadow compare 或强制 graph 返回
- 路由 debug 字段补充
- answer guard
- no-answer 时清空 final context / citations
- MySQL 持久化 message 和 `message_context`

所以要记住：

> `RAGService.query()` 负责“服务层外围治理”，不是只转发到 pipeline。

## 4. `RAGPipeline.ask()` 到底负责什么

代码位置：

- [pipeline.py](/D:/System/Desktop/regurag/backend/app/rag/pipeline.py:438)

它是**核心问答流水线**，主要做 6 步。

### 4.1 组装 `QueryPolicy`

入口先把本次请求的查询策略收口成 `QueryPolicy`：

- retrieval 用什么策略
- `top_k_retrieve`
- `top_k_rerank`
- `enable_rerank`
- `answer_style`

对应：

- [query_policy.py](/D:/System/Desktop/regurag/backend/app/rag/query_policy.py:9)

### 4.2 准备查询输入

如果 service 层已经提前准备好了 `query_prep`，这里直接复用；
否则自己调用 `prepare_query_inputs()`：

- query rewrite
- alias 扩展
- query keywords 提取
- 拼出 `search_query`

### 4.3 调统一 `Retriever`

核心调用点：

- [pipeline.py](/D:/System/Desktop/regurag/backend/app/rag/pipeline.py:487)

这里已经不是自己手搓 dense + sparse 了，而是统一走：

- `self._get_retriever().retrieve(...)`

返回内容包括：

- `query_vector`
- `retrieved_docs`
- `supplemental_docs`
- `deduped_docs`
- `valid_docs`
- `doc_embeddings`
- `retrieved_debug_chunks`

### 4.4 MMR 和 parent 聚合

拿到 `valid_docs` 之后，继续做：

- MMR 选 chunk
- policy question 判断
- 保留关键规则 chunk
- 保留 query keyword 命中的 chunk
- 聚合成 `parent_docs_for_rerank`

### 4.5 rerank 和 final context

如果启用 rerank：

- 用 `Reranker` 对 parent docs 打分
- 排序
- 构建最终 `final_context_parents`

### 4.6 LLM 生成和 debug 构建

最后：

- 拼 context
- 调 `llm.generate(...)`
- 构建 citations
- 构建 debug

所以一句话：

> `RAGPipeline.ask()` 负责核心问答流水线，但不负责路由、短路和持久化。

## 5. `Retriever` 这一层现在怎么理解

最容易看不懂的，通常就是这一层。

先只记职责，不要先记所有类名。

### 5.1 现在为什么单独有 `Retriever`

因为以前的 dense + sparse + merge + dedupe + sort 逻辑散在：

- `RAGPipeline.ask()`
- shadow workflow 的 `retrieve_documents_node()`

现在已经收口成统一 retrieval core。

### 5.2 当前结构

主要看这几个文件：

- [hybrid_retriever.py](/D:/System/Desktop/regurag/backend/app/rag/retrievers/hybrid_retriever.py:14)
- [policy.py](/D:/System/Desktop/regurag/backend/app/rag/retrievers/policy.py:7)
- [vector_store.py](/D:/System/Desktop/regurag/backend/app/rag/vector_store.py:13)
- [__init__.py](/D:/System/Desktop/regurag/backend/app/rag/retrievers/__init__.py:15)

职责这样理解最省脑：

- `VectorStore`
  - 负责向量写入
  - 负责 dense search
  - 负责维护当前 sparse provider 所需索引

- `HybridRetriever`
  - 负责把 dense / sparse 的结果拼起来
  - 负责 dedupe / sort / debug chunk

- `RetrievalPolicy`
  - 决定这次 retrieval 怎么跑
  - 比如 `dense_top_k / sparse_top_k / sparse_min_hits / enable_sparse`

### 5.3 当前的 sparse provider

现在支持：

- `sqlite_fts`
- `bm25`
- `scan`
- `none`

默认是 `sqlite_fts`。

这里要记住：

> “混合检索”没有变，变的是“关键词检索的实现方式已经工程化了”。

以前是 Python 全量扫描。
现在默认是 `SQLite FTS5` 本地全文索引。

## 6. 影子链路现在覆盖到哪

影子图入口：

- [graph.py](/D:/System/Desktop/regurag/backend/app/workflows/rag/graph.py:13)

当前只有 4 个节点：

1. `prepare_query`
2. `retrieve_documents`
3. `select_parents`
4. `generate_answer`

对应文件：

- [nodes.py](/D:/System/Desktop/regurag/backend/app/workflows/rag/nodes.py:25)

### 6.1 它覆盖的是哪一段

它覆盖的是：

- query prep
- retrieval
- MMR / parent selection
- rerank / final context / generate

也就是**核心 pipeline**。

### 6.2 它没覆盖什么

这些还在 `RAGService.query()` 外围：

- 知识库自动路由
- 会话解析
- 轻意图 / FAQ / off-topic 短路
- cross-domain guard
- answer guard
- MySQL 持久化

所以当前 shadow graph 的定位是：

> 核心 RAG pipeline 的影子实现，不是完整服务链影子实现。

---

## 7. 哪些地方是主链路和影子链路共用的

这一点很重要，因为现在并不是两套完全独立实现。

当前共用的主要有：

- `rewriter`
- `query_alias_expander`
- `vector_store`
- `reranker`
- `llm`
- `retrieval_rules`
- `Retriever`
- 大部分 `pipeline_steps` 工具函数

也就是说：

> 现在主链路和影子链路，已经共用同一套 retrieval core 和大部分核心步骤函数。

它们的差别主要在于：

- 一个是直接写在 `ask()` 里顺序执行
- 一个是拆成 graph node / workflow state 执行

## 8. 现在应该怎么读代码

不要再按“模块目录从头翻到尾”的方式读。

推荐阅读顺序：

1. 先读 [rag_service.py](/D:/System/Desktop/regurag/backend/app/services/rag_service.py:684)
   只看 `query()`，先看哪里 short-circuit，哪里真正进入 pipeline

2. 再读 [pipeline.py](/D:/System/Desktop/regurag/backend/app/rag/pipeline.py:438)
   只看 `ask()`，把 query prep -> retrieve -> mmr -> rerank -> generate 串起来

3. 再读 [hybrid_retriever.py](/D:/System/Desktop/regurag/backend/app/rag/retrievers/hybrid_retriever.py:14)
   只看 `retrieve()`，理解 dense/sparse 合并

4. 再读 [vector_store.py](/D:/System/Desktop/regurag/backend/app/rag/vector_store.py:13)
   看 dense search 和 sparse provider 分发

5. 最后再读 [nodes.py](/D:/System/Desktop/regurag/backend/app/workflows/rag/nodes.py:25)
   把它当成 `ask()` 的拆分版去看

如果还要继续深入，再看：

- `ConversationContextResolver`
- `IntentRouter`
- `KnowledgeBaseRouter`
- `answer_guard`

## 9. 看不懂时先问自己哪 3 个问题

每次读不懂，不要直接在 helper 之间跳。

先问：

1. 我现在看的这段，属于服务层外围，还是核心 pipeline？
2. 这段是在“准备输入”、还是“做检索”、还是“做结果治理”？
3. 这段是主链路专属，还是主链路和 shadow 共用？

只要先回答这 3 个问题，代码会容易读很多。

---

## 10. 当前一句话心智模型

如果只保留一个最简模型，就记这个：

- `RAGService`：外围治理和持久化
- `RAGPipeline`：核心问答流水线
- `Retriever`：统一召回层
- `shadow graph`：核心流水线的对照实现

也就是：

> 外围在 service，核心在 pipeline，召回在 retriever，对照在 shadow graph。
